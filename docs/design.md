# 本地视频转写设计

日期：2026-07-21

## 1. 目标

在当前 M4 Max、36GB 统一内存的 Mac 上建立一个可重复执行的混合工作流：输入本地 MP4，由 Whisper 在本机生成带时间戳的原始 JSON，再由 Gemini 2.5 Flash 在线完成英文翻译、中文润色、标题识别和章节整理，最终只交付一个结构化中文 JSON。

示例输入文件：`/path/to/video.mp4`。

现场探测结果：视频时长 3485.95 秒（约 58 分 06 秒），H.264 视频、AAC 音频，1280×720，文件约 238MB。

## 2. 范围

本期包括：

- 本地视频音轨提取。
- 中文和英文自动语言识别。
- 原始 Whisper JSON 中间产物。
- 英文转中文，中文做纠错、去口语冗余和排版润色。
- 自动生成总标题、摘要、章节标题和段落。
- 最终 JSON 保留原始分段 ID 与时间范围，便于回查。
- 用一个命令执行完整工作流。
- 仅在本机 `127.0.0.1` 运行的浏览器工作台：选择视频、查看进度、预览 Markdown、下载、复制和导入 Obsidian。
- 本机设置页：切换 Gemini / Ollama 的受支持模型、配置非敏感参数，并通过 macOS Keychain 管理 Gemini 凭据。

本期不包括：

- B 站在线下载、账号登录或绕过访问限制。
- 说话人分离、声纹识别和逐词卡拉 OK 字幕。
- 远程服务、多人协作、账号体系和局域网访问。
- 将 SRT、TXT 或烧录视频作为默认交付物。

## 3. 技术方案

### 3.1 音频预处理

使用 Homebrew 安装的 `ffmpeg`，从 MP4 提取 16kHz、单声道、16-bit PCM WAV。原视频保持只读，中间音频写入项目的 `work/`。

### 3.2 语音识别

使用 Homebrew `whisper-cpp` 和 `ggml-large-v3-turbo.bin`：

- 模型约 1.62GB，当前硬件有充分余量。
- `-l auto` 自动识别中文或英文。
- 只生成 Whisper 完整 JSON，保留检测语言、分段文本和时间戳。
- 默认使用 GPU；启用 Flash Attention，失败时回退到兼容参数。

选择未量化的 Large v3 Turbo，而不是 Q5 量化版，优先保证混合中英文、专有名词和嘈杂背景下的识别质量。后续只有在实际性能不足时才评估 Q5。

### 3.3 翻译、润色与标题整理

- 本地程序先将连续 Whisper 分段组成约 160 字符的固定来源组，再将来源组打包为不超过 6000 字符的处理块；Gemini 为保证逐组约束服从度，每块额外最多 4 个来源组；Ollama 回退不设该组数上限。
- 默认调用 Gemini Developer API 的稳定模型 `gemini-2.5-flash`，温度设为 0，并将 `thinkingBudget` 设为 0；不启用搜索、工具或会话存储。
- 使用 Gemini Structured Outputs 和 JSON Schema 约束每块返回结构，本地程序仍执行语义校验，不能把结构化输出当作内容正确性的保证。
- 中文内容：修正明显 ASR 错字和标点，删除不影响语义的重复、语气词，整理为自然段。
- 英文内容：翻译成简体中文，同时执行相同的润色和段落整理；必要的英文专有名词可保留。
- 每块必须返回纯 JSON，包含块标题、摘要和与固定来源组一一对应的按序段落。每段限制为 80–260 字、2–3 句；本地验证失败时仅重试一次。程序只接受模型返回的固定来源组 ID，再由原始 JSON 精确回填该组的源分段 ID 与起止时间；发现来源组缺失、乱序或重复即拒绝结果。原文跨时间的重复观点不作为串组证据。
- 最后只把各块的标题和摘要交给模型，再生成一个总标题、总摘要并统一章节标题，避免重复发送全文。
- 不确定的人名、术语或数字标记为 `【待核对】`，不得凭空补充事实。

### 3.4 输出结构

```text
work/<video-name>/
├── audio.wav
└── transcript.whisper.json

outputs/<video-name>/
├── transcript.final.json
└── run.log
```

`transcript.final.json` 同时包含来源信息、模型信息、检测语言、总标题、摘要、章节、段落、源分段映射和运行耗时，不再单独维护 `run.json`。

核心结构：

```json
{
  "schema_version": "1.0",
  "source": {
    "path": "/absolute/input.mp4",
    "duration_seconds": 3485.95,
    "detected_language": "zh"
  },
  "models": {
    "asr": "whisper-large-v3-turbo",
    "editor": "gemini-2.5-flash"
  },
  "title": "视频总标题",
  "summary": "内容摘要",
  "sections": [
    {
      "id": 1,
      "title": "章节标题",
      "start_ms": 0,
      "end_ms": 120000,
      "paragraphs": [
        {
          "start_ms": 0,
          "end_ms": 30000,
          "source_segment_ids": [0, 1, 2],
          "text": "润色后的中文段落"
        }
      ]
    }
  ],
  "run": {
    "started_at": "ISO-8601",
    "finished_at": "ISO-8601",
    "elapsed_seconds": 0
  }
}
```

## 4. 安全与数据边界

- 原视频、音频、Whisper 模型、原始 Whisper JSON 和最终 JSON 保留在本机。
- 只有按块整理后的 Whisper 文本、固定来源组 ID 和处理指令发送给 Gemini；不上传视频、音频、绝对文件路径或本地时间戳。
- API Key 只从 `GEMINI_API_KEY` 环境变量读取，不写入代码、配置、日志、缓存或最终 JSON。
- 免费层数据策略与付费层不同；当前目标是公开 B 站内容，涉及私密或内部视频时不得使用免费层。
- 默认使用 Gemini；保留 `TRANSCRIPT_PROVIDER=ollama` 显式回退能力，不修改 Ollama、Claude Code 或其他客户端配置。
- 不复制、覆盖或删除 Downloads 中的源视频。
- `models/`、`outputs/` 和 `work/` 通过 `.gitignore` 排除。

## 5. 主要风险与处理

| 风险 | 处理方式 |
|---|---|
| 背景音乐或长静音导致幻觉 | 保留原始 Whisper JSON，最终段落可按源分段回查 |
| 中英文混说 | 对每个文本块按内容处理，不只依赖视频级语言标签 |
| 模型破坏时间轴 | 模型只返回源分段 ID，时间戳由本地程序回填 |
| 模型返回结构合法但内容错误 | 本地校验来源组覆盖、顺序、长度和句数；失败重试一次后终止，并保留原始 Whisper JSON 供人工抽查 |
| 专有名词识别错误 | 标记 `【待核对】`，不在首版引入术语库 |
| Key 缺失、无效或额度不足 | 启动前检查环境变量；对 401/403 明确失败，对 429 和 5xx 做有限重试 |
| API Key 泄露 | 请求使用 Header 传递；异常信息和日志不得输出请求 Header 或 Key |
| 中途失败 | 每个处理块独立缓存，可从已有 Whisper JSON 继续执行 |

## 6. 验收标准

- 一条命令可处理指定 MP4。
- `transcript.final.json` 可被标准 JSON 解析器读取，且不含 Markdown 包裹。
- 所有章节和段落时间合法、单调且不超出视频时长。
- 每个原始 Whisper 分段恰好被最终段落引用一次，无大段遗漏或重复覆盖。
- 最终段落以简体中文为主；英文内容已翻译，必要专有名词除外。
- 开头、中段、结尾至少抽查 5 个片段，润色未改变原意或编造事实。
- 总标题、摘要和章节标题能概括内容，且最终 JSON 包含完整运行元数据。
- 运行日志能定位依赖、模型、转写或翻译阶段的失败。
- 日志、缓存、最终 JSON 和 Git 状态中均不存在 Gemini API Key。

## 7. 本地浏览器工作台

### 7.1 定位与视觉方向

工作台是个人 Mac 上的本地浏览器应用，不是云端后台。它服务于一个完整动作：选择一个视频，知道它正在经历什么，完成后舒适地阅读文章并交给下一处知识库。

已选定视觉方向为“安静的编辑台”：暖白纸张底色、墨色正文、低饱和玉绿色作为唯一强调色；文章预览而不是仪表盘占据主视觉。侧栏提供“转写工作台”“历史记录”和“设置”三个入口。正文使用 macOS 系统的苹方 / 宋体回退，避免在线字体请求；正文行宽不超过约 65 个汉字。

### 7.2 页面与交互

**转写工作台**

- 空状态只呈现一个主操作“选择本地视频”。浏览器不能安全取得文件系统绝对路径，因此此操作调用后端的 macOS 原生文件选择器；不使用 `<input type=file>` 上传，也不复制源视频。
- 已选择状态显示从源视频实际截取的封面、文件名、时长、格式和当前模型；“开始转写”是唯一主按钮。服务使用 `ffmpeg` 在约 10% 时长处截取一帧（限制在 8–60 秒），缓存为对应输出目录的 `cover.jpg`；它只读取源视频，缓存命中时不重复提取。
- 运行状态使用五步时间线：音频提取、Whisper 识别、Gemini 整理、结构校验、生成 Markdown。当前阶段有明确文字、百分比、Gemini 块号和可展开日志；未完成时不展示未经校验的半成品 Markdown。
- 完成状态把 Markdown 渲染为长文阅读面，保留标题、源视频引用、摘要、章节和段落。阅读区右侧提供可滚动的章节大纲，章节按钮与预览中的标题锚点一一对应，点击后定位到相应内容；下载、复制和导入 Obsidian 操作置于大纲下方。复制成功和导入成功均给出短暂、可感知的反馈。
- 失败状态显示可读的失败阶段和脱敏错误，不显示密钥或完整请求；可在不重新选择视频的前提下重试。

**历史记录**

- 服务扫描所有 `outputs/*/transcript.final.json`，以输出目录名作为稳定记录 ID，按完成时间（缺失时按文件修改时间）倒序展示；这使 CLI、Web 和基准测试产生的既有结果都可被查看，且同名视频不会相互覆盖。
- 每条记录显示真实视频封面、标题、源文件名、时长、所用模型和完成时间；点击记录进入与完成状态一致的 Markdown 阅读面、章节大纲及下载、复制、导入操作。
- 历史 API 不返回源视频绝对路径。视频仍在本机且没有缓存封面时，可按需生成 `cover.jpg`；视频已移走且没有缓存封面时使用中性视频图标，不以其他图片冒充封面。

**设置**

- 提供方仅允许 Gemini 和本机 Ollama。Gemini 可选受支持模型、思考配置和温度；Ollama 只列出 `127.0.0.1:11434` 实际可用模型，不允许填写远程端点。
- Gemini Key 以“未配置 / 已配置（脱敏）”呈现；保存、替换和删除都只操作 macOS Keychain。模型和非敏感偏好写入应用支持目录，不落在 Git 仓库。
- Obsidian 初次使用时调用原生目录选择器选择 Vault，并配置一个可选子目录。导入时以转写标题生成 Markdown；同名文件自动递增编号，不覆盖已有笔记。

### 7.3 本地架构与数据流

```text
浏览器页面
  ├── GET / API：任务、历史记录、设置、Markdown、封面与下载
  ├── SSE：任务阶段、块级进度和日志事件
  └── 原生选择器请求
        ↓
本机 Python Web 服务（仅 127.0.0.1）
  ├── 任务运行器：启动既有 transcribe-video.sh
  ├── 设置服务：应用配置 + macOS Keychain
  ├── Markdown 服务：调用既有 export-transcript-markdown.py
  ├── 历史服务：索引 outputs 下的已验证最终 JSON 与派生封面
  └── Obsidian 服务：写入已确认的 Vault 目录
        ↓
ffmpeg / whisper.cpp / Gemini 或 Ollama / JSON 校验
```

服务一次只运行一个转写任务。运行器为每个任务在对应输出目录写入可恢复的本地状态；服务重启后，已有 `transcript.final.json` 的任务标记为已完成，其余标记为已中断并允许重试。输出目录新增派生文件 `transcript.final.md` 和可选的 `cover.jpg`；二者都从本地证据派生，不替代 JSON 作为时间轴证据。

为避免把日志文本当作 API，编排脚本和文本处理脚本新增单行 JSON 进度事件，页面通过 SSE 订阅。日志仍保留现有可读文本，事件中不得包含视频绝对路径、Gemini Key 或请求正文。

### 7.4 配置与安全边界

- 非敏感设置位于 `~/Library/Application Support/Local Video Transcriber/settings.json`，包括提供方、模型、思考配置、温度、Obsidian Vault 与子目录；不记录 Key。
- Gemini Key 使用 `security` 命令存入 macOS Keychain。启动子进程时临时注入 `GEMINI_API_KEY`，以兼容现有脚本；进程结束后不写回文件或日志。
- 服务固定绑定 `127.0.0.1`，不提供登录和多用户能力；本机用户拥有完整控制权。
- 原视频仍只读；只有用户明确选择的 Obsidian Vault 目标可以写入 Markdown。下载和复制不修改源文件。

### 7.5 前端验收标准

- 浏览器选择视频不会上传或复制源视频，且只能选择本地普通视频文件。
- 进度在页面刷新前实时更新；完成、失败与中断三个状态均可被准确呈现。
- 预览、下载和复制的内容均来自同一份 `transcript.final.md`。
- 封面可由当前源视频实际截取；历史记录只在源视频不可用且无缓存时显示中性图标，绝不使用无关图片替代。
- 历史记录能读取既有 `outputs/*/transcript.final.json`，同名视频或基准结果按输出目录独立呈现，且历史 API 不泄露源视频绝对路径。
- 章节大纲与 Markdown 标题一一对应，点击任一章节可准确定位到正文，且不影响下载、复制和导入的一致性。
- 导入 Obsidian 后，Vault 中存在内容一致的 Markdown，且绝不覆盖同名笔记。
- 设置页不会显示、写入或记录 Gemini Key；本机 Ollama 不存在或模型不存在时有明确提示。
- 在 1440×1024 和常见笔记本宽度下，长文预览、主按钮与运行状态均可读、可操作，键盘焦点清晰。

## 8. Gemini 网络恢复与实时进度

### 8.1 问题结论

浏览器任务在 Gemini 整理阶段停留在固定百分比，并非 Whisper 或 Gemini 推理本身持续占用计算资源。现场证据表明：Whisper 已完成，至少一个文本块已通过 Gemini 并写入块级缓存；随后进程在 Gemini 官方端点的 IPv6 TCP 建连阶段处于 `SYN_SENT`。同一端点可经 IPv4 正常建立连接。

现有标准库 `urllib` 请求把连接、上传和读取响应共用 600 秒总超时。IPv6 路由不可达时，单次请求会长时间无输出；编排层只在整块完成后发送进度事件，浏览器因此只能停留在上一块的百分比，无法区分“模型正在生成”和“网络没有连接成功”。

### 8.2 已选调用路径

- Gemini 继续使用 `generateContent`，每个块等待一份完整、受 JSON Schema 约束的结果。该任务是可等待完整结果的离线文本整理，不改用面向多轮代理的 Interactions、聊天式流输出或非即时 Batch。
- Gemini Python 调用从手写 `urllib` REST 请求迁移到官方 `google-genai` SDK；SDK 通过同步 HTTPX 客户端发送请求。新增依赖只服务于官方客户端、连接超时与重试控制。
- Gemini 请求显式约束到 IPv4 本地地址，避免系统优先尝试当前不可用的 IPv6 路径；仍保留环境代理支持，不写入或猜测代理配置。
- 每个请求采用独立的连接、写入和读取超时：连接约 8 秒、写入约 30 秒、读取约 120 秒。连接异常不再占用 600 秒；对确实开始生成但较慢的响应仍保留合理读取窗口。
- 仅对网络超时、429、408 和 5xx 等瞬态异常进行有限的指数退避和随机抖动；认证、参数、Schema 和内容校验错误立即失败，不做无意义重试。

### 8.3 可恢复性与进度

- 块级 JSON 缓存的结构、版本和校验语义保持不变。已验证的块可以直接复用；修复后的同一任务从未缓存块继续，不重新提取音频或运行 Whisper。
- 每个块发起请求前发送明确进度；请求尚未返回时，每 5 秒发送一次不含文本或密钥的等待心跳，包含块号、总块数、当前尝试次数和已等待秒数。
- 重试、成功、最终失败均产生可读的结构化进度事件。Web 服务原样转发这些事件到 SSE，使页面能显示“连接中 / 等待响应 / 第几次重试”，而不是长期停在静态百分比。
- 任务取消或因超时失败后，已完成的缓存与原始 Whisper JSON 保留；最终 JSON 仍只在全量校验成功后原子替换。

### 8.4 安全与边界

- 继续只向 Gemini 发送按块组织的字幕文本、来源组 ID 和整理指令；不发送视频、音频、绝对路径、时间戳或完整 Whisper JSON。
- Key 仍只来自 `GEMINI_API_KEY` 或浏览器启动时的 Keychain 临时注入。日志、SSE、异常和测试替身不得输出 Key、请求 Header 或请求正文。
- 旧 Ollama 回退路径不修改；它仍只访问 `127.0.0.1:11434`。

### 8.5 验收标准

- 在 IPv6 不可达、IPv4 可达的本机环境中，Gemini 调用不再尝试不可恢复的 IPv6 长连接；连接异常在约 8 秒后进入受控重试或明确失败。
- 运行中的浏览器任务至少每 5 秒收到一次 Gemini 阶段更新，且能显示块号、尝试次数或等待时长。
- 复用已有 Whisper JSON 和已验证块缓存后，恢复任务不会重新执行 ffmpeg 或 Whisper。
- 模拟 401/403、429、5xx、连接超时和非法结构化响应时，重试策略、失败信息和 Key 脱敏均符合边界。
- 完整任务通过原有 JSON 验证与 Markdown 派生验证；来源映射、时间范围和最终原子替换语义不变。

## 9. 固定阅读器与交付工具栏

### 9.1 阅读布局

完成态与历史阅读复用同一套固定阅读器。桌面端结果区域由正文阅读框与文章大纲组成：正文阅读框使用稳定的视口相关高度，Markdown 正文在框内独立滚动；视频信息、进度时间线和页面其他区域不因长文阅读而持续拉长。章节定位只滚动正文阅读框到对应标题，不再调用整页滚动。

正文阅读框保留独立的顶部栏。顶部栏包含 Markdown 状态和轻量阅读设置：展示宽度提供“紧凑 / 标准 / 宽屏”三档，字号提供“小 / 标准 / 大”三档。设置仅改变正文列宽和排版字号，使用浏览器 `localStorage` 保存，不写入转写结果、应用设置或后端。

### 9.2 完整大纲与交付操作

右侧栏只承担文章大纲。目录不再设内部最大高度或滚动条，章节以略紧凑的字号和间距自然完整展示；标题较长时允许换行，不截断、不省略。大纲在桌面端与正文阅读框顶部对齐。

“导入 Obsidian”“下载 Markdown”“复制全文”从右侧栏移至阅读区域上方的横向交付工具栏。工具栏置于阅读框和大纲之前，明确展示“文稿交付”及本地安全说明；主要操作保持 Obsidian 导入，下载与复制为并列次操作。所有入口仍使用同一份已验证的 `transcript.final.md`。

### 9.3 响应式与验收

- 宽屏桌面端：正文框固定高度并内部滚动，大纲完整显示且无内部滚动；交付操作横向排列。
- 窄屏端：结果区改为单列，正文仍在独立阅读框滚动；大纲与交付操作按正常文档流纵向排列，避免固定高度挤压可读性。
- 章节按钮定位后，对应标题在正文阅读框内可见，页面整体滚动位置不因章节定位改变。
- 宽度与字号调整即时生效，刷新页面后仍保持；Markdown 内容、下载、复制和 Obsidian 导入不受显示偏好影响。

## 10. 夜间阅读模式

### 10.1 交互与持久化

夜间模式是浏览器工作台的显示偏好，不属于转写配置，也不写入应用支持目录、最终 JSON 或 Markdown。侧栏主导航中增加一个带月亮 / 太阳图标的切换按钮：桌面端显示当前可执行操作的文字，窄屏端与其他导航项一样仅保留图标，并通过 `aria-label` 和 `aria-pressed` 保持可访问性。

默认继续使用现有浅色主题。用户切换后，前端立刻在根元素写入主题标识，并通过浏览器 `localStorage` 保存；重新打开或刷新工作台时优先恢复该偏好。主题切换不请求后端、不影响运行中的任务、SSE 连接、阅读宽度字号偏好或已生成文稿。

### 10.2 视觉范围

夜间模式沿用“安静的编辑台”而不是纯黑高对比界面：页面背景使用深墨绿灰，阅读纸面使用略亮的深森林色，正文采用暖灰白，玉绿色仍仅用于进度、选中态和主要操作。侧栏、卡片边界、表单控件、正文预览、章节大纲、历史列表、Toast、错误状态及键盘焦点都使用同一组主题变量，避免出现浅色残留或正文层级消失。

浅色主题的现有视觉不改变。两套主题都要保留长文正文、二级标题、引用、辅助说明和禁用态之间可辨识的层级；预览框滚动条与封面缺失态也必须随主题协调变化。

### 10.3 验收标准

- 任意页面均能通过同一按钮即时在浅色和夜间模式间切换，图标、文案和辅助状态正确反映当前可执行操作。
- 刷新后恢复用户上一次选择；首次访问仍是浅色模式。
- 工作台、历史记录、设置页和固定阅读器在两套主题下均无明显浅色残留，正文与控件可读、焦点可见。
- 切换主题不会重新加载 Markdown、改变阅读宽度字号，也不会中断正在执行的转写任务。

## 11. 一键启动浏览器工作台

日常入口使用项目根目录的 `start-workbench.sh`。现有 `scripts/start-web.sh` 保持为前台开发与排障命令：它构建前端并直接运行 Uvicorn，终端持续显示日志，可用 `Ctrl+C` 停止。新入口不替代、不修改其职责。

`start-workbench.sh` 先向 `http://127.0.0.1:${PORT:-8765}/api/settings` 发起本机健康检查。服务已在运行时，只打开现有控制面板，不重建前端、不创建第二个服务；端口被非本项目进程占用时明确失败，绝不结束该进程。服务未运行时，脚本在 `work/` 中创建本机临时日志，后台调用现有前台脚本，轮询健康检查成功后使用 macOS `open` 打开浏览器。

启动失败或等待超时必须显示日志末尾以便排查。端口、绑定地址和数据边界不改变：服务仍只监听 `127.0.0.1`，不上传视频或新增任何远程能力。

验收：首次执行一条命令即可启动并打开工作台；再次执行只复用现有服务；冲突端口不被杀死；依赖或启动错误可从终端与临时日志定位。
